# v0.10.3 — PATCH release scope

Consolidates everything that landed on `main` after `v0.10.2` and
identifies the items that are still deferred past this release. The
actual code for this patch is already merged; this plan exists so
there's a single document to read before cutting the tag and to
explicitly park the items that aren't in it.

## Context

Between v0.10.2 (2026-04-22) and the writing of this plan (2026-04-23),
three inputs fed into the work:

1. A technical audit (Agent A) and a release-readiness audit (Agent B)
   run in parallel on `d154b41` and synthesised in
   `hypomnema-v10.2-comprehensive-audit.md`. Converged on one CRITICAL
   (`install.sh` missing 3 hook registrations) and two HIGH
   (`incrementRecurrence` race, bus factor — non-code).

2. A Theory of Functional Systems audit (TFS, Anokhin) in
   `hypomnema-tfs-audit.md` — surfaced the "open quanta" problem: a
   large fraction of sessions write inject events but never get
   posterior classification, biasing the Bayesian `eff` metric.

3. A CI failure on `main` that bisected to an incomplete drain of
   backgrounded hook children before test-fixture cleanup (only
   `memory-fts-shadow.sh` was waited for; six other children were
   missed).

## Deferred past v0.10.3

Items that surfaced during this work and are not in the tag. Each
line calls out its source so the decision to defer stays auditable.

### Retroactive silent classification in `wal-compact.sh`

**Source:** TFS audit + author's measurement — `docs/notes/measurement-
bias.md` §"What might help later" item 1.

On the author's corpus, 312 injects in the post-v0.7.1 window have no
posterior classification because the owning sessions never reached
Stop. Current Bayesian `eff = (pos+1)/(pos+neg+2)` measures on ~25% of
the inject volume. Proposed fix: `wal-compact.sh` walks injects older
than 24h with no closing event on their session_id and emits
`trigger-silent` retroactively, restoring the denominator.

Not in v0.10.3 because the design decisions are not obvious:

- Should retroactive `outcome-positive` also be emitted for mistakes
  that didn't repeat within a follow-up window? (Different time
  horizon than trigger classification.)
- How to tag retroactive events — a `retro:` prefix in the target?
  A sixth column? (Sixth column violates FORMAT.md §5.2 four-column
  invariant — same blocker as session-metrics fix below.)
- Interaction with `memory-analytics.sh`'s rolling 30-day window for
  `eff` computation.

Effort once designed: ~2 hours. Needs a design note first.

### `session-metrics` wire-format fix

**Source:** found during TFS audit analysis (not either external
audit). Documented in `docs/notes/measurement-bias.md` §"Side
finding".

`hooks/session-stop.sh:450` writes:

    TODAY|session-metrics|DOMAINS|error_count:X,tool_calls:Y,duration:Zs

but FORMAT.md §5.2 declares `date|event|target|session_id` immutable
within format v1. Column 4 here is metrics pairs, not session_id —
the one defined event that violates the layout. Consequence: no way
to join `session-metrics` to `inject` events by session_id, which is
what #1 above needs for a correct "stop ran for this session" signal.

Fix requires a format v1 → v2 bump — adding a column, reinterpreting
an existing one, or moving metrics into the target-pairs slot
alongside a dedicated session_id column. All three are breaking for
readers, which is why this is parked as its own release concern
rather than patched inline.

### Secrets detector on PreToolUse for memory writes

**Source:** Agent A MEDIUM (security) + Agent B ethical FAQ gap.

Nothing today stops a user from writing `password: hunter2` in a
mistake body — it lands in `additionalContext` every session and in
`.tfidf-index` / `index.db`. Proposed: PreToolUse hook with a
whitelist regex (`\b(api[_-]?key|secret|password|token|aws[_-]?...)\s*[:=]`)
that blocks the write and prints an actionable warning.

Effort: 1-2 hours code, plus meaningful time on false-positive
calibration (a regex-only list will reject legitimate code that
happens to contain those tokens). Not worth the time-slice inside
this patch given the benefit/cost ratio.

### `bench-memory.sh` CI integration

**Source:** v0.10.2 plan deferred list (`docs/plans/2026-04-22-v0.10.2.md`).

`make bench` exists, prints wall-clock numbers for each hook hot
path, no thresholds. Adding wall-clock gates in CI catches
regressions but requires a tolerance design — CI runners vary in
speed, fixed timeouts produce flakes. Deferred pending a design
for percentile-based thresholds or relative-to-baseline comparison.

### Go coverage gaps

**Source:** Agent A coverage table + same v0.10.2 deferred list.

- `internal/fts` — 8.3% (only `tokenize_test.go`; index/shadow/sync
  untested at unit level).
- `cmd/memoryctl/` — 0% (no test file).

Mechanical: write unit tests. Not in v0.10.3 because the right shape
of `cmd/memoryctl` tests — full subprocess-and-env roundtrips vs
internal-function tests — wants a separate decision.

### Decompose `session-start.sh` / `session-stop.sh`

**Source:** Agent A MEDIUM + LOW in the code-quality section.

786 lines and 610 lines respectively. v0.10.2 plan already said
"partial decomposition exists (`evidence-extract.sh`)". Pure-refactor
work, no behavioural change, low-leverage right now — deferred until
a specific bug or new feature forces touching those files. Breaking
them up preemptively just to shrink line count produces churn without
payoff.
