# v0.11 — Decisions autoreflection

## Context

ADR в `decisions/` содержат `## When to revisit` секции — критерии
пересмотра решения (precision ниже N, shadow-miss корреляция выше M,
календарная дата). Сейчас это проза, которую никакой механизм не
читает автоматически. Если `measurable_precision` в `self-profile.md`
упадёт с 66% до 35%, ни один хук и ни один subcommand не укажет «ADR
`substring-triggers-with-negation.md` сейчас под давлением, пора
открыть ревью».

Это gap в self-awareness петле проекта: `self-profile.md` агрегирует
метрики, `memory-strategy-score.sh` синхронизирует `success_count` у
стратегий из WAL, `mistake.recurrence` растёт автоматически на
дубликатах — но ADR висят в тишине и стареют до тех пор, пока
случайный агент не прочитает их вручную и не сравнит глазами.

## Scope

- Добавить structured `review-triggers:` поле во frontmatter ADR.
- Реализовать `memoryctl decisions review` — evaluator, сопоставляющий
  триггеры с `self-profile.md` и WAL-метриками.
- Добавить секцию «Decisions under pressure» в `self-profile.md`.
- Обратно заполнить (backfill) триггеры для трёх существующих ADR.

## Non-goals

- Автоматическое принятие решений. Мы сигналим «revisit», не
  выключаем ADR и не переписываем.
- Семантический анализ тела ADR. Только frontmatter-метрики.
- LLM в цикле review. Deterministic metric eval только.
- Уведомления вне `self-profile.md` / CLI output.

## Design

### Frontmatter schema

```yaml
review-triggers:
  - metric: measurable_precision
    operator: "<"
    threshold: 0.40
    source: self-profile
  - metric: shadow_miss_ratio
    operator: ">"
    threshold: 0.30
    source: wal
  - after: "2026-07-22"     # calendar fallback; no metric/operator/source
```

Supported metrics (initial set):

| Metric | Source | Computed from |
|---|---|---|
| `measurable_precision` | self-profile | `(useful + silent-applied) / (useful + silent)` |
| `recurrence_top_mistake` | self-profile | max recurrence across `mistakes/` |
| `shadow_miss_ratio` | wal | `shadow-miss events / total prompts` over last N days |
| `ref_count` | file frontmatter | own file's `ref_count` |

Operators: `<`, `<=`, `>`, `>=`, `==`, `!=`.

Calendar triggers use `after: YYYY-MM-DD` — no `metric`/`operator`,
just a date. Fires if `today >= after`.

### Command

```
$ memoryctl decisions review
[pressure] fts5-shadow-retrieval — shadow_miss_ratio = 0.34 > 0.30
[overdue]  bash-go-gradual-port — after 2026-07-22 passed (14 days ago)
[ok]       substring-triggers-with-negation
```

Exit code: 0 if all ok, 1 if any pressure/overdue.

### Integration with self-profile

New section appended to `self-profile.md`:

```markdown
## Decisions under pressure

- **fts5-shadow-retrieval** — shadow_miss_ratio = 0.34 (threshold 0.30)
- **bash-go-gradual-port** — review-by 2026-07-22 passed
```

Empty section omitted when no ADRs flagged.

## Implementation steps

1. Extend frontmatter parser (`internal/memory/frontmatter.go`) to
   accept `review-triggers:` as a block-sequence of heterogeneous
   maps. Preserve forward-compat: unknown fields ignored.
2. `internal/decisions/eval.go` — pure metric evaluator. Accepts a
   `Snapshot` struct (self-profile metrics + WAL aggregates) and a
   trigger list, returns pass/fail per trigger.
3. `internal/decisions/snapshot.go` — snapshot builder. Reads
   `self-profile.md`, computes `shadow_miss_ratio` from WAL over a
   fixed window (proposed: 14 days).
4. `cmd/memoryctl/decisions.go` — `review` subcommand wiring.
5. Teach `memoryctl self-profile` to append «Decisions under pressure»
   section after running `decisions review` internally.
6. Unit tests per metric + golden tests for `review` output.
7. Backfill `review-triggers:` in three current ADRs.
8. Update `CLAUDE.md` (frontmatter schema section) and `README.md`
   (decisions subsection) to document the new field.
9. Update `docs/CONFIGURATION.md` to remove the «not yet implemented»
   caveat.

## Acceptance

- `make test` passes.
- `memoryctl decisions review` on seeded WAL with synthetic
  `measurable_precision < 0.40` returns at least one `[pressure]`
  finding and exit code 1.
- `memoryctl self-profile` includes «Decisions under pressure»
  section when any ADR triggers.
- Three existing ADRs parse successfully with structured triggers.

## Open questions

- **Source syntax.** Enum (`self-profile`, `wal`, `file`) or
  free-form string? Enum is stricter but adds validation code; free-form
  lets users invent metrics with zero schema churn. Proposal: enum
  initially, relax if users need custom sources.
- **WAL window for `shadow_miss_ratio`.** Proposed 14 days — rolling
  window balancing responsiveness vs noise on small session counts.
  Should it be configurable? Proposed: hardcode initially, tune after
  first dogfood month.
- **Missing-metric behaviour.** If `self-profile.md` doesn't carry the
  requested metric (older format, field renamed): warn, skip, or fail?
  Proposal: skip with WAL `decision-review-metric-missing` event.
- **Qualitative triggers.** ADR прозой говорит «if users complain about
  trigger-authoring» — не автоматизируется. Leave in body as prose, or
  mark with `review-triggers: - qualitative: "..."` для документации?
  Proposal: prose only, structured field is for mechanical metrics.
