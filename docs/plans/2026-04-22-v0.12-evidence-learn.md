# v0.12 — Evidence learn

## Context

Evidence phrases in `mistakes/`/`feedback/`/`knowledge/` files are
authored by hand. For a new user this creates cold-start: during the
first 20-30 sessions, seed files ship with EN-only evidence that
doesn't match the user's natural output style (Russian, Japanese,
mixed-language, or just personal idioms). Result — silent rules,
zero recall, `measurable_precision` starts low until evidence is
hand-tuned.

Concrete example from dogfood (session 2026-04-22): my personal
`wrong-root-cause-diagnosis.md` had 9 academic-register evidence
phrases ("гипотеза", "альтернативные объяснения"). Measured result —
13 `trigger-silent` events, 0 `trigger-useful`, despite 231 inject
events. Expanding evidence with 19 natural-speech phrases (validated
against three real outcome-negative session JSONLs) recovered one
silent session into useful and added 5+ matches per clean session,
with 0 false-positives against two noise probe sessions.

The manual tuning loop is valuable but slow. Automating it from
existing session JSONL transcripts turns cold-start from a 30-session
problem into a one-time bootstrap command.

## Scope

- New subcommand `memoryctl evidence learn`.
- Read user session JSONLs under `~/.claude/projects/*/*.jsonl`.
- For each target file (subset of `mistakes/` + `feedback/` specified
  by the user, or all pinned rules), propose candidate evidence
  phrases extracted from the user's actual assistant text.
- Interactive approval — user sees each candidate with counts and
  decides accept / reject / always-reject-pattern.
- Output: updated frontmatter `evidence:` blocks.

## Non-goals

- Automatic evidence updates without user approval. Evidence
  authoring is a trust-critical operation — false-positive phrases
  silently corrupt precision metrics.
- LLM-based phrase extraction. Deterministic n-gram + noun-phrase
  extraction only, to keep the tool offline and auditable.
- Cross-user evidence sharing or aggregation. Personal voice does not
  transfer between users.
- Phrase translation. If a user writes in Russian, the tool extracts
  Russian phrases — it does not suggest English equivalents.

## Design

### Command

```
memoryctl evidence learn [--target SLUG] [--sessions N] [--dry-run]

--target SLUG    # one or more mistake/feedback slugs; default = all pinned rules
--sessions N     # last N session JSONLs to scan; default = 30
--dry-run        # print proposed phrases without writing
```

Example interaction:

```
$ memoryctl evidence learn --target wrong-root-cause-diagnosis

Scanning 27 sessions (2026-03-24 to 2026-04-22)…
Target: wrong-root-cause-diagnosis (current evidence: 9 phrases)

Silent sessions: 7
For each silent session, found candidate enumeration phrases in assistant text:

  [9 sessions]  "варианты:"         — ACCEPT / reject / reject-pattern?
  [7 sessions]  "shortlist"         — ACCEPT / reject / ...
  [5 sessions]  "мой голос"         — accept / REJECT / ... (already in my list)
  [5 sessions]  "три варианта"      — ACCEPT / reject / ...
  [12 sessions] "(1)"               — accept / reject / REJECT-PATTERN
                                       (too broad — matches any numbered list)

Candidates adopted: 3
Writing to ~/.claude/memory/mistakes/wrong-root-cause-diagnosis.md …
evidence grew 9 → 12 phrases.
```

### Mining algorithm

For each target rule, for each session where the rule was silent
(`trigger-silent` in WAL, rule was injected, but no evidence phrase
matched):

1. Extract assistant text from session JSONL (lowercase, strip code
   fences, strip quoted content to match session-stop.sh behaviour).
2. Detect enumeration-style content using three signals:
   - Sequences of `(N) ...` or `N. ...` markers within a 500-char
     window.
   - Lines starting with `- ` or `* ` in groups of 2+.
   - Table rows with "Вариант", "Option", "Approach", "Candidate"
     headers.
3. Within detected enumeration regions, extract candidate phrases:
   - 2-gram and 3-gram tokens (case-insensitive).
   - Noun phrases near enumeration markers ("my candidates", "три
     варианта").
4. Filter:
   - Phrase must appear in ≥20% of silent sessions for this target.
   - Phrase must appear in ≤10% of non-silent sessions (would
     otherwise be too generic).
   - Phrase length ≥5 characters (avoid "(1)", "№1").
   - Phrase not already in current evidence (no duplicates).
5. Rank by silent-session coverage, present top 15.

### Frontmatter update

When the user accepts candidates, append them to `evidence:` block
preserving order and indentation. Do not touch other frontmatter
fields (no unnecessary diff noise).

### Rejection memory

`memoryctl evidence learn` persists rejected phrases per target in
`~/.claude/memory/.runtime/evidence-rejections/<slug>.list`. Subsequent
runs skip them. User can empty the file to reconsider.

## Implementation steps

1. `internal/jsonl/reader.go` — stream-parse JSONL (some sessions
   are >1MB); extract `.message.content[].text` where
   `type="assistant"`.
2. `internal/evidence/extract.go` — enumeration detector + n-gram
   extraction + filter thresholds.
3. `internal/evidence/rank.go` — silent-coverage vs non-silent-noise
   rank.
4. `cmd/memoryctl/evidence.go` — `learn` subcommand wiring,
   interactive REPL, rejection list.
5. Frontmatter writer — append-only update to `evidence:` block.
6. Unit tests on synthetic JSONLs + golden comparison against hand-
   tuned evidence from dogfood.
7. Docs: README section «Bootstrap evidence for your voice»;
   CONFIGURATION.md cross-reference.

## Acceptance

- `make test` passes.
- On the dogfood session corpus (the 3 outcome-negative + 5 clean
  sessions used to validate wrong-root-cause manual tuning),
  `memoryctl evidence learn --target wrong-root-cause-diagnosis
  --dry-run` proposes at least 10 of the 19 phrases that were
  manually validated in 2026-04-22, with 0 false-positives against
  the 2 noise probe sessions.
- REPL accepts / rejects interactively without crashing on empty
  JSONLs or malformed session files.

## Open questions

- **Multi-language mining.** Should the tool detect the user's
  dominant language from JSONL and bias toward it, or always
  extract any n-gram regardless of script? Proposal: extract
  all, rank by silent-coverage only — language emerges from the
  data without explicit detection.
- **Pinned-only scope.** Should non-pinned rules be eligible for
  mining? They might be too transient for evidence tuning.
  Proposal: `--target` flag required for non-pinned; `--all` runs
  only on pinned by default.
- **False-positive protection after first acceptance.** If the
  user accepts "(1)" and later regrets it, how does the tool
  recover? Proposal: `memoryctl evidence review <slug>` shows
  precision per phrase from WAL and suggests removals.
- **Seed files.** Should `seed: true` files be excluded from
  evidence learn? They are format examples for new users, not
  personal memory. Proposal: yes, skip seeds — evidence learn is
  for owned files.
