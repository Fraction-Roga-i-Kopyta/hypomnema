# Roadmap: v0.10.5 → v1.0

Consolidated sequence for the work that's currently split across
`CHANGELOG.md § Deferred to v0.10.5+`,
`docs/plans/2026-04-22-v0.11-decisions-autoreflection.md`,
`docs/plans/2026-04-22-v0.12-evidence-learn.md`, and
`docs/notes/measurement-bias.md § What might help later`.

No calendar dates — bus factor is 1, dates are fiction. Order and
dependencies only.

## Versioning strategy

The format contract (`docs/FORMAT.md`, `docs/hooks-contract.md`) stays
at `v1` through the entire 0.10.x → 0.13.x line. Everything until v1.0
is either backward-compatible addition or localised internal fix.

- **PATCH** (0.10.5, 0.10.6 …) — bugfix, refactor, test coverage.
  Visible behaviour unchanged; `MIGRATION.md` untouched.
- **MINOR** (0.11, 0.12, 0.13) — new subcommand, new frontmatter
  field, new WAL event type. Existing files and WAL entries keep
  working unchanged; new features are opt-in.
- **MAJOR** (1.0) — format v1 → v2 bump. Breaking WAL grammar change
  (session-metrics fix is the forcing function). `MIGRATION.md` gets
  a v1→v2 section with an automated migrator.

## Dependency graph

```
v0.10.5 ─┬─► v0.11 (Decisions autoreflection)
         │
         ├─► v0.12 (Evidence learn) ─────────┐
         │                                    │
         └─► v0.13 (Retroactive silent +      │
                    Secrets detector) ────────┤
                                              ▼
                                        v1.0 (format v2 + public)
```

v0.10.5 decomposes the two largest hooks, unblocking clean additions
in v0.11 and v0.12. v0.11 and v0.12 are independent — either can ship
first in practice; the numbering follows the existing plan documents.
v0.13 consumes signals v0.12 produces (fewer silent events → smaller
retroactive-silent queue) but doesn't strictly depend on it. v1.0
requires everything earlier to be on `main` because the format bump
needs a stable reference point to diff against.

## v0.10.5 — PATCH · Decomposition + coverage

Pure refactor and mechanical test coverage. No behavioural change;
every test that passed in v0.10.4 must pass byte-for-byte.

### Scope

1. **Extract `hooks/session-stop.sh` (608 lines) into `lib/`:**
   - `lib/outcome-detection.sh` — evidence match + outcome-positive /
     outcome-negative logic.
   - `lib/rotation.sh` — stale → archive lifecycle.
   - `lib/reminder-builder.sh` — strategies / checkpoint reminder
     text assembly.
   - `evidence-extract.sh` stays where it is (already extracted).
   - After: session-stop.sh is the dispatch skeleton only, ≤200 lines.

2. **Extract `hooks/session-start.sh` (785 lines) into `lib/`:**
   - `lib/detect-domains.sh` — the existing inline `detect_domains()`.
   - `lib/health-check.sh` — the three-warning health block.
   - `lib/collect-scored.sh` — scored-pool collection + filters.
   - After: session-start.sh ≤400 lines, main body is sequence of
     library calls.

3. **`cmd/memoryctl/` Go coverage 0% → ≥50%:**
   - Subprocess tests for each subcommand's happy path and one error
     path. Use `exec.Command` against the built binary + fixture
     memory dirs; don't invoke `main()` internally.
   - Cover: `fts shadow`, `fts sync`, `fts query`, `dedup check`,
     `self-profile`, `doctor`, `tfidf rebuild`.

4. **`bench-memory.sh` CI integration** (if percentile-threshold
   design lands — otherwise defer):
   - Makefile `bench-ci` target that invokes `bench-memory.sh` and
     parses output against `docs/performance-budgets.md`.
   - Budgets are percentile-of-baseline, not absolute ms — tolerant
     to CI-runner variance.

### Acceptance

- `make test` 262/262 unchanged.
- `make parity` 6/6 unchanged.
- `wc -l hooks/session-*.sh` drops below 600 combined.
- `go test -cover ./cmd/memoryctl/` ≥ 50%.
- No CHANGELOG migration notes required.

### Why this release exists

Every v0.11+ feature currently has to thread new behaviour through
600-line hook files. Decomposing first is the kind of preemptive
refactor the v0.10.3 plan explicitly deferred — but the deferral was
conditional on "no feature needs to touch those files". v0.11 and
v0.12 both touch session-stop.sh (review-triggers integration with
self-profile, evidence-learn reading injected-slugs lists). That
condition has now fired.

## v0.11 — MINOR · Decisions autoreflection

Referenced plan: `docs/plans/2026-04-22-v0.11-decisions-autoreflection.md`.

### Scope (summary — full detail in plan)

- `review-triggers:` structured frontmatter field on ADRs. Metric +
  operator + threshold + source. Four initial metrics:
  `measurable_precision`, `recurrence_top_mistake`,
  `shadow_miss_ratio`, `ref_count`. Calendar trigger via `after:
  YYYY-MM-DD`.
- `memoryctl decisions review` — reads ADRs, evaluates triggers
  against `self-profile.md` and WAL aggregates, prints a
  pressure/overdue/ok list. Exit 1 if any trigger fires.
- `self-profile.md` gains `## Decisions under pressure` section when
  review runs as part of Stop hook flow.
- Backfill `review-triggers:` for the 8 ADRs shipped in v0.10.4 —
  each has a prose "When to revisit" section, translate the metric
  ones into structured form.

### Why MINOR, not PATCH

New frontmatter field (`review-triggers:`) even though additive — any
new scanner-visible field crosses the threshold. Existing ADRs
without the field behave exactly as before.

### Acceptance

- All 8 ADRs have `review-triggers:` where a structured form is
  expressible. Purely calendar / purely prose cases stay prose-only.
- `memoryctl decisions review` on the author's install reports at
  least one "pressure" case (the WAL is ripe for this).
- `self-profile.md § Decisions under pressure` renders on the next
  Stop-hook generation.
- `memoryctl doctor` gains a check that reports count of pressure ADRs.

### Dependencies

- v0.10.5 session-stop decomposition (review runs inside the Stop
  hook flow; touching an already-decomposed file is cheap).
- None on v0.12.

## v0.12 — MINOR · Evidence learn

Referenced plan: `docs/plans/2026-04-22-v0.12-evidence-learn.md`.

### Scope (summary — full detail in plan)

- `memoryctl evidence learn [--target SLUG] [--sessions N] [--dry-run]`
  — interactive subcommand.
- Reads session JSONLs under `~/.claude/projects/*/*.jsonl`.
- For each silent target rule, mines 2/3-gram noun-phrase candidates
  from assistant text in silent sessions. Deterministic extraction
  (no LLM in the loop).
- Interactive approval loop: accept / reject / reject-pattern per
  candidate.
- Writes accepted phrases to `evidence:` frontmatter blocks.
- Persists rejections per-slug in `.runtime/evidence-rejections/`.

### Why MINOR

New subcommand + new runtime directory + behaviour (writes to
frontmatter). Not a format change — the `evidence:` field has
existed since v0.7.1.

### Acceptance

- `make replay` on the author's corpus shows measurable lift in
  `measurable_precision` post-evidence-learn run (baseline captured
  in v0.10.4; expect +10+ percentage points on at least one target).
- `trigger-silent` events per inject drop by ≥20% on targets that
  got ≥3 new phrases accepted.
- `docs/notes/measurement-bias.md` gets a follow-up entry quantifying
  the fraction of open quanta that are now closing because silent
  classification actually fires.

### Dependencies

- v0.10.5 decomposition (shares `lib/evidence-extract.sh` parsing).
- None on v0.11.

## v0.13 — MINOR · Retroactive silent + Secrets detector

Two independent items; bundled into one release because both are
small and both build on observability foundations from v0.10.4.

### Scope: Retroactive silent classification

Unblocks the measurement bias documented in
`docs/notes/measurement-bias.md`. Sessions older than a threshold
(probably 24 h, configurable via `WAL_RETROACTIVE_AGE`) with inject
events but no closing signal receive a synthesised
`trigger-silent-retro|<slug>|<session>` WAL event per inject.

Design decisions to make before implementation:

- **Event naming.** Proposed `trigger-silent-retro` (new type, so
  it's countable separately in analytics and doctor). Alternative:
  reuse `trigger-silent` with a `[retro]` suffix in target — rejected
  because it violates "target is a slug" convention.
- **Where it runs.** `wal-compact.sh` (already runs periodically,
  has WAL lock). Alternative: new `memoryctl wal reclassify` manual
  command — rejected because manual invocation defeats the
  auto-correction purpose.
- **What about `outcome-positive`.** Mistakes that didn't repeat in
  a session whose Stop never ran should plausibly earn positive
  outcomes. But the timing is tricky (≥7 days without the mistake
  occurring, not 24 h). Out of scope for this release; document as
  a v0.14 follow-up.

### Scope: Secrets detector on PreToolUse

Current PreToolUse hook only runs dedup on `mistakes/*.md`. Add a
second matcher for memory-file writes that scans for secret patterns
before the write lands on disk. Initial pattern set:
`\b(api[_-]?key|secret|password|token|aws[_-]?(?:access|secret)[_-]?key)\s*[:=]\s*\S+`.

False-positive mitigation:

- Only scan files under `~/.claude/memory/` (the user is
  intentionally authoring memory there).
- Only block if the match is outside a fenced code block (code
  examples of API keys in documentation are fine).
- Exit 2 with an actionable message pointing at the matched line.
- `HYPOMNEMA_ALLOW_SECRETS=1` env bypass for the one-off legitimate
  case.

### Acceptance

- Author's corpus analytics report gains a row for
  `trigger-silent-retro` in `## Trigger usefulness`.
- `doctor open_quanta_last_30d` ratio drops substantially (we've
  measured 93% open pre-retro; expectation ≤40% post-retro, since
  only truly-crashed sessions should remain open).
- A fixture test tries to write a memory file with `api_key: AKIA…`
  and gets blocked with the actionable stderr message.

### Dependencies

- v0.10.5 decomposition (secrets detector hook goes through
  `lib/` helpers for memory-dir path resolution).
- Soft dependency on v0.12: retroactive-silent's target list is
  smaller if evidence-learn has already converted many would-be
  silents into useful events. Shipping v0.13 before v0.12 works;
  the retroactive queue is just larger.

## v1.0 — MAJOR · Format v2 + public release

The only release in this sequence that's genuinely breaking. Gated
behind one technical necessity (session-metrics fix) and one social
one (at least one external user has run the system end-to-end for
two weeks).

### Scope: Format v2 bump

`docs/FORMAT.md` version moves from `1` to `2`. The changes are:

- **`session-metrics` column 4 = session_id.** Metrics pairs move
  to column 3 structured form: `session-metrics|domains:backend,duration:120s,tool_calls:15|session_id`.
  Existing v1 WAL entries continue to parse (readers detect the
  format by presence of `session_id`-looking column 4).
- **Event type registry.** `docs/EVENTS.md` becomes normative —
  every WAL event has a typed schema for its third column. Enforced
  by a new `memoryctl wal validate` subcommand that refuses unknown
  types by default (tolerant mode for forward-compat).
- **Optional 5th column for contextual metadata.** Strictly for
  new event types; v1 events stay four-column immutable. Version
  marker in a comment header of `.wal` (`# hypomnema wal v2`).

### Scope: Public release prep

- `CONTRIBUTING.md` with "PRs by invitation until v1.1, issues open".
- `SECURITY.md` with vulnerability-report channel.
- Repo visibility → public.
- Show HN / r/ClaudeAI draft from `docs/maintenance/DISTRIBUTION_DRAFTS.md`,
  revised with v1.0 numbers from actual external usage.
- `make replay` baseline published with a 2-week dogfood corpus.

### Non-goals

- Not building v2 → v1 downgrade. Once format v2 is written, rollback
  means re-running migration backwards; out of scope.
- Not changing any ranking formula constants. Those stay tunable
  via `.config.sh` but their defaults are the same as v0.10.4.
- Not migrating seed files en masse. New seeds ship with v2
  frontmatter (explicit `format_version: 2`); v1 seeds keep working
  under backward-compat reader.

### Acceptance

- At least one external user has installed v0.13.x, used it for two
  weeks of real work, and filed either a bug report or an improvement
  suggestion. This is the social gate — "hypomnema works for me"
  doesn't count.
- Format v2 migrator (`memoryctl migrate v1 v2`) converts a v0.10.4
  corpus to v2 shape with zero data loss; round-trip parity test.
- `MIGRATION.md § v0.13 → v1.0` documents the upgrade flow, including
  the automated migration command.
- Clean-VM bootstrap on both ubuntu:24.04 and macOS-fresh-HOME pass.
- GitHub Release entry created (see RELEASE_CHECKLIST §5.1 — the
  miss we made twice already).

### Dependencies

- All prior releases on `main`.
- External user signal (not controllable by the author).

## Items explicitly NOT in this roadmap

- **LLM in the retrieval loop.** Stays out. The moat is
  deterministic, auditable retrieval; adding an LLM would dissolve it.
- **Team / multi-user memory sharing.** Out of scope for 1.x. If a
  user wants shared memory, they `git push` to a shared remote.
- **Vector embeddings.** Out of scope permanently (decision in
  `docs/decisions/file-based-memory.md`).
- **Native Anthropic memory API adapter.** Watched, not built. When
  Claude Code ships a primary memory surface, hypomnema becomes a
  compatibility shim — that's a v2.0 reshape, not a 1.x task.
