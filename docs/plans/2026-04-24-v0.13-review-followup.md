# Followup: external review v0.13 (2026-04-24)

## Контекст

Источник: `hypomnema-v0.13-deep-review.md` — внешнее ревью от 2026-04-24, собранное 5 параллельными агентами по срезам (Go-ядро, bash-хуки и v0.13 фичи, scoring/математика, документация/UX, реальный корпус живого пользователя «Артёма»). В приложении ревью явно перечислены контрольные пересечения (несколько находок подтверждены двумя независимыми агентами), что повышает доверие к диагнозу.

Ревью верифицировано автором: все 3 P0 и 4 из 4 проверенных P1 подтверждаются прямым чтением кода с точностью до строки. Данные корпуса Артёма (32 файла, 14 дней WAL, 0 mistakes, 4% trigger-useful) и моего собственного корпуса (93 файла, 29 mistakes, 4.4% trigger-useful) сходятся в одном: **scoring-формула на реальных корпусах работает не так, как обещает CLAUDE.md**.

Этот план — дорожная карта работ, вытекающих из ревью. Он **не исполняется немедленно** — сначала обсуждение с ревьюером (см. `docs/audits/2026-04-24-v0.13-review-response.md`), потом отдельный commit на каждый блок.

## P0 — release 0.13.1

Три фикса, один тест-регрессия на каждый. Все обратимы, все ≤15 строк.

### P0.1 Edit bypass в `memory-secrets-detect.sh`

- **Файл:** `hooks/memory-secrets-detect.sh:35`
- **Регистрация:** `install.sh:260` — `PreToolUse Write|Edit`
- **Факт:** хук читает только `jq -r '.tool_input.content // empty'`. `Edit` tool кладёт текст в `.tool_input.new_string`. Все Edit-операции в `~/.claude/memory/` проходят мимо гейта.
- **Live-evidence** (из ревью, § 2.1): `{"tool_name":"Edit","tool_input":{"new_string":"password: ABCDEFGH12345678","file_path":"...memory/feedback/test.md"}}` → `exit 0`.
- **Фикс:**
  ```bash
  CONTENT=$(printf '%s' "$INPUT" | \
    jq -r '.tool_input.content // .tool_input.new_string // empty' 2>/dev/null)
  ```
- **Regression test:** в `hooks/test-memory-hooks.sh` добавить два теста:
  - `Edit` с `new_string: "api_key: LONGRealTokenValue12345"` → `exit 2`;
  - `Edit` с `new_string: "# just plain docs, no secret"` → `exit 0`.
- **Risk:** низкий. Изменение — просто добавляет второй jq-fallback. Прежнее Write-поведение не затронуто.

### P0.2 `uninstall.sh` оставляет 5 из 8 хуков в `settings.json`

- **Файл:** `uninstall.sh:111-114`
- **Факт:** хардкод pair-list перечисляет только 3 хука:
  ```
  SessionStart:memory-session-start.sh
  Stop:memory-stop.sh
  UserPromptSubmit:memory-user-prompt-submit.sh
  ```
  `install.sh:256-263` регистрирует **8** хуков. Разница (5 штук): `memory-dedup`, `memory-secrets-detect`, `memory-outcome`, `memory-error-detect`, `memory-precompact`.
- **Симптом после uninstall:** симлинки удалены (`_remove_our_symlinks` это честно делает), но `settings.json` ссылается на призраков. Каждый `Write/Edit/Bash/PreCompact` будет сыпать «no such file» в Claude Code.
- **Фикс:** вывести pair-list **из одного источника правды** с `install.sh`. Варианты:
  1. Общая константа в `hooks/lib/hook-registry.sh` — читается и `install.sh`, и `uninstall.sh`.
  2. Парсить `register_hook` вызовы из `install.sh` awk'ом внутри `uninstall.sh`.
  3. Динамически читать текущие команды, содержащие `~/.claude/hooks/memory-*.sh`, из `settings.json` — и удалять их все.
  
  Предпочитаемо (3) — устойчиво к будущим добавлениям хуков, не требует синхронизации двух мест.
- **Regression test:** новый bash-тест `uninstall-removes-all-hook-entries.sh`:
  1. Seed fake `settings.json` с 8 hook-entries.
  2. Run `uninstall.sh`.
  3. Assert: `jq '.hooks | [.. | .command? | strings] | map(select(contains("memory-"))) | length' settings.json == 0`.
- **Risk:** низкий. uninstall — одноразовая операция.

### P0.3 Race `wal-retro-silent` vs `wal-compact` в Stop fan-out

- **Файл:** `hooks/session-stop.sh:333-352`
- **Факт:** оба скрипта запускаются через `&` без `wait` между ними. Оба честно дерутся за общий WAL-lock — но **порядок захвата не определён**.
- **Зависимость:** комментарий на `:333-335` явно предупреждает:
  > Must run BEFORE compaction because compaction turns `inject` into `inject-agg` and loses session_id.
  
  Но `&` эту зависимость не обеспечивает.
- **Что сломается, если compact победит:** `wal-retro-silent.sh:60` в pass 2 матчит `$2 == "inject"`. После compaction эти события уже `inject-agg` — session_id потерян, retro ничего не найдёт. Фича v0.13 тихо перестаёт работать на следующей же сессии.
- **Фикс (минимальный):**
  ```bash
  # retro — синхронно, сразу ждём
  CLAUDE_MEMORY_DIR="$MEMORY_DIR" bash "$RETRO_SCRIPT" 2>/dev/null
  # compact — фоново, как было
  CLAUDE_MEMORY_DIR="$MEMORY_DIR" bash "$COMPACT_SCRIPT" 2>/dev/null &
  ```
  Retro быстрый (бенч из ревью: ~250ms на 100K строк WAL), блокирующий Stop на четверть секунды — приемлемо.
- **Regression test:** новый тест в `test-memory-hooks.sh`:
  1. Seed WAL со старым `inject` без closing event (cutoff > 24h).
  2. Run `session-stop.sh`.
  3. Assert: в WAL появилась строка `trigger-silent-retro|<slug>|<session>`.
  4. И только после retro — `inject-agg` (результат compact'а).
- **Risk:** низкий. Stop latency +~250ms на 100K WAL; для типичных WAL (<5K строк) незаметно.

## P1 — release v0.14

### P1.0 Cold-start scoring — activation ADR `proposed → active`

- **ADR:** `docs/decisions/cold-start-scoring.md` (создан 2026-04-24 по итогам round 2 external review).
- **Суть:** Bayesian eff и TF-IDF body match **gated** за min-signal порогами (defaults `MIN_OUTCOME_SAMPLES=5`, `OUTCOME_WINDOW_DAYS=14`, `MIN_TFIDF_VOCAB=100`, все ENV-overrideable). При недостаточном сигнале Bayesian возвращает **1.0 нейтрально** (не 0.5), TF-IDF молчит.
- **Видимость обязательна:**
  - `memoryctl doctor` — строка `[INFO] scoring components active: N/5 (…dormant: Bayesian (need ≥5 outcomes in 14d, have 0), TF-IDF (vocab <100))`.
  - `_agent_context.md` — параллельная пометка для субагентов.
  - `session-stop.sh` — ежедневный `scoring-mode|<active_count>|<dormant_list>` в WAL.
- **Точки реализации:**
  - `hooks/lib/score-records.sh` — sample-count gate перед Bayesian формулой.
  - `internal/tfidf/tfidf.go` (или `session-start.sh:373`) — vocab-count gate.
  - `internal/doctor/doctor.go` — новый check `scoring_components_active`.
  - `hooks/session-stop.sh` — daily emit.
  - `CONFIGURATION.md` — секция `§ Scoring cold-start gates`.
- **Калибровка перед `proposed → active`:** замер на **synthetic корпусах в репо** (primary) + **локальной копии external-01** у мейнтейнера (gut-check, не в CI). Критерий принятия defaults:
  - На `synthetic-mature` (15 mistakes, 30 outcome-positive) — Bayesian должен быть active для всех относящихся slug'ов.
  - На `synthetic-cold-start` (0 outcomes) — Bayesian должен быть dormant для 100% slug'ов.
  - На моём локальном корпусе (29 mistakes, 40 outcome-positive) — Bayesian active для >30% slug'ов.
  - На локальной копии external-01 (0 outcomes) — dormant для 100% slug'ов (подтверждение, что synthetic-cold-start реалистичен).
  - Если synthetic и real расходятся — править `MIN_OUTCOME_SAMPLES` или `OUTCOME_WINDOW_DAYS`, не фиксить кодом.
- **Regression tests:**
  - Unit: Bayesian с <5 outcomes → 1.0; с ≥5 → формульный результат.
  - Integration: doctor output матчит формат dormant-строки.
  - Fixture (primary, в CI): `synthetic-cold-start` → `active: 3/5`; `synthetic-mature` → `active: 5/5`.
  - Local gut-check (не в CI): external-01 → `active: 3/5` (ожидаемо совпадает с synthetic-cold-start verdict).
- **Merge criterion:** ADR `status: active` только после того, как synthetic fixtures зелёные И локальный cross-reference с external-01 дал совпадающий verdict. Запись в `docs/fixtures/real-corpus-notes.md`.
- **Risk:** средний. Главный побочный — «тёмные тестовые корпуса» (синтетические fixtures) могут начать вести себя иначе. Проверить `test-memory-hooks.sh` на cold-start не-регрессию.

### P1.1 `outcome-positive` за пределы `mistakes/`

**Главный системный фикс всего ревью.** Без него `measurable precision` остаётся в диапазоне 1-5% у любого пользователя, чей корпус не mistake-centric. У меня самого — 40 outcome-positive на 2027 inject (все из mistakes/).

- **Файл:** `hooks/lib/outcome-detection.sh:61`
- **Корень:** жёсткий хардкод `MISTAKE_FILE="$MEMORY_DIR/mistakes/${mistake_name}.md"`.
- **Дизайн-решение:**
  - обобщить функцию на все типы, где есть evidence-block (мы умеем его майнить из JSONL в v0.12);
  - параметризовать не именем типа, а **наличием mineable evidence**: если у файла есть `evidence:` в frontmatter или если `evidence-extract` смог вытащить фразы — пробовать match в транскрипте;
  - evidence-match положительный → emit `outcome-positive|<slug>|<session>` (тот же WAL-формат, что сегодня для mistakes).
- **Зависимости:** использовать уже существующую логику из `hooks/lib/feedback-loop.sh` (feedback detector с evidence-match) — не писать второй раз. **Блокирован P1.0** — cold-start ADR должен активироваться до, иначе нельзя чисто замерить эффект P1.1.
- **Тесты:** 3 теста — feedback с evidence → outcome-positive; knowledge без evidence → ничего (не регрессия); mistake по-старому.
- **Scope inflation risk:** легко расползётся в «а давайте измерять выборочно по доменам / по recency». **Не надо.** Сохраняем единый критерий: есть evidence-match в транскрипте = outcome-positive.
- **Калибровочный эксперимент — merge criterion:**
  - Primary (в CI): прогон на `synthetic-feedback-heavy` с synthetic transcripts. Ожидание: PRE-P1.1 → 0 outcome-positive для feedback-слагов; POST-P1.1 → 4 outcome-positive (ровно столько транскриптов содержат evidence-фразы). Control: 5-й транскрипт без evidence не эмитит outcome-positive.
  - Primary (в CI): прогон на `synthetic-mature` — existing outcome-positive для mistakes НЕ ломается (regression).
  - Secondary (local gut-check): прогон на моём корпусе — ожидаемая дельта `measurable precision` +5..+15 п.п.
  - **Stop-condition:** если `synthetic-feedback-heavy` не эмитит ровно ожидаемые 4 outcome-positive — не merge'ить, разобраться (evidence-mining не работает на feedback-контенте, или feedback-match логика не переиспользуется корректно).
  - Локальный gut-check на external-01 после merge — проверка, что real feedback-корпус тоже получает positive эмиссии (sanity, не блокер merge).

### P1.5 `wal-retro-silent` — closing-set расширение + soft-close semantics + corrupt-guard

- **Файл:** `hooks/wal-retro-silent.sh:50`
- **Закрытый set сейчас:** `trigger-useful`, `trigger-silent`, `outcome-positive`, `outcome-negative`, `clean-session`, `trigger-silent-retro`.
- **Корень проблемы (из round 2):** `clean-session` смешивает две семантики — «Stop отработал» и «сессия прошла без errors». У Артёма 17% сессий clean при фактических 100% сессий-в-которых-Stop-отработал. Это раздувает `trigger-silent-retro` ложными «молчаниями».
- **Решение v0.14 — soft-close semantics в FORMAT.md:**

  > A session is considered "closed" for retroactive silent
  > classification if its WAL contains any event from the closing
  > set: `clean-session` (zero errors, terminal), `session-metrics`
  > (Stop hook completed, non-zero errors acceptable), `outcome-new`
  > (fresh mistake captured), `outcome-positive` / `outcome-negative`
  > (evidence-matched injected slug), `trigger-useful` (explicit
  > reference), `trigger-silent` (natural silent).

- **Фикс в `wal-retro-silent.sh:50`:** расширить closing-set на `session-metrics` + `outcome-new`. Остальные события (`error-detected-LEVEL`, `cluster-load`, `cascade-review`, `strategy-used`, `strategy-gap`, `evidence-missing`, `evidence-empty`) — **не добавляем**: они описывают состояние ВНУТРИ сессии, не её закрытие. Сабвопрос: есть ли в реальных корпусах сессии с `error-detected-*` без `session-metrics`? Если >0 — вторая диагностика (error-detect срабатывает без Stop'а).
- **Corrupt-WAL guard:** `NF == 4` в pass 2, плюс quoting-safe awk — не падаем на битых строках.
- **Regression tests:** синтетическая сессия с `inject + session-metrics` (без `clean-session`) → после retro НЕ должна появиться `trigger-silent-retro`.
- **Docs:** `docs/FORMAT.md` — новая секция `## Closing events (soft-close semantics)`.

## Open questions — статус после round 2

После обмена с Артёмом (см. `docs/audits/2026-04-24-v0.13-review-response-round-2.md`) большая часть вопросов разрешена. Ниже — трек статуса.

### 1. Чинить Bayesian + TF-IDF или упростить scoring? — **РАЗРЕШЁН (variant C)**

**Наблюдение:** на корпусе Артёма (0 mistakes) и на моём (29 mistakes, но всё равно 4.4% trigger-useful) — `wal_spaced_repetition` и `tfidf_body_match` вносят минимальный сигнал. По факту работают только `keyword_hits × 3` и `project_boost × 1`.

**Принятое решение:** гибрид — **gating + fix, последовательно**.

- **Cold-start ADR (P1.0)** обрабатывает «как формула ведёт себя, пока сигнала мало» — runtime self-disable с видимостью.
- **P1.1 (outcome-positive expansion)** обрабатывает «как данные собираются для feedback-heavy корпусов» — без этого Bayesian у feedback-only пользователей остаётся dormant навсегда.

ADR лёг в `docs/decisions/cold-start-scoring.md` (status: proposed). Merged → active после калибровки defaults на обоих корпусах.

Спасибо, Артём — это был мой пробел (не задать вопрос «а нужна ли формула в этом виде»). 5 агентов смотрят внутри компонентов, не задают «нужен ли компонент».

### 2. Whitelist для `memory-secrets-detect`: хардкод или config? — **РАЗРЕШЁН (`.secretsignore`)**

- `.secretsignore.default` в репо (committed, `seeds/**` по умолчанию, не редактируется пользователем, overwritten at upgrade).
- `.secretsignore` в `~/.claude/memory/` (опциональный, пользовательский, `.gitignore`-семантика: line-by-line glob, `#` комменты, `!` negation, `**`).
- Detail — в P1.6 плана.

### 3. Делать ли `triggers:` обязательным для `feedback`/`knowledge`? — **РАЗРЕШЁН (`doctor WARN`, не FAIL)**

- `doctor WARN` для файлов `feedback`/`knowledge` без `triggers:`. С hint'ом «consider adding triggers:» — опциональный, но видимый.
- Шаблоны подсвечивают `triggers:` как рекомендуемое, не обязательное.
- Никакого `memoryctl lint --require-triggers` — не enforcing, слишком строго для хрупкой адопции.

### 4. Эскалация `open quanta last 30d = 83%` — **РАЗРЕШЁН (soft-close)**

Из round 2: `clean-session` пишется только при 0 errors, что **hook-controlled** (пользователь не может «закрыть сессию правильно» — решает хук). 83% open quanta у Артёма = нормальное свойство сетапа, где сессии имеют errors > 0 (обычный workflow разработки).

Решение: расширить `closing-set` в `wal-retro-silent.sh:50` на `session-metrics` + `outcome-new` — это soft-close семантика. Детали в P1.5, docs — `FORMAT.md § Closing events`.

Сабвопрос на замер: сколько сессий имеют `error-detected-*` без `session-metrics`? Если >0 — вторая диагностика (error-detect срабатывает в сессиях, где Stop-хук не отработал).

### 5. Темп релизов — **остаётся открытым**

За 14 дней — 20 релизов. v0.12.0 → v0.12.1 в тот же день («recovery from tracking gap»).

**Что решено в round 2:** добавить corpus-based CI на `fixtures/corpora/synthetic-*/`. Это не лечит «темп», но лечит «уверенность»: каждый тег должен пройти doctor + self-profile + injection-diff на synthetic суите. Локальный gut-check на external-01 — rare manual, не блокирующий. Триггер для многоагентного audit — только hook-contract / scoring-changes изменения, чтобы не превратить в cargo cult.

**Открыто:** стоит ли ввести минимальный cooldown между minor-релизами (например, 48 часов)? Нет однозначного ответа — «медленнее» не всегда значит «лучше», особенно когда патч-скорость — реальный способ отвечать на внешние ревью.

## Corpora для тестирования — synthetic-first

Изначально план предполагал коммит external-01 snapshot в `fixtures/corpora/external-01/`. После повторного размышления решение **пересмотрено**: реальный пользовательский snapshot **не коммитится** в публичный репо.

**Причины отказа от external-01 commit:**
- Residual leakage risk (manual scan требуется, не проводился).
- Imitation risk (перекошенный onboarding-корпус может стать каноном для новичков).
- Bus factor: git-история не стирается.
- **Критично:** snapshot не содержит транскриптов сессий — P1.1 evidence-match на нём невозможен.

**Принятое решение:** `fixtures/corpora/synthetic-*/` — 5 параметризованных синтетических профилей с `expected.json` snapshot-based assertions. Детальная спека: `docs/plans/2026-04-24-synthetic-fixtures.md`.

Синтетические профили:
- `synthetic-cold-start` — блокирует P1.0 калибровку (Bayesian + TF-IDF dormant);
- `synthetic-mature` — baseline всех 5 компонентов active;
- `synthetic-feedback-heavy` — блокирует P1.1 validation (evidence-match для non-mistakes, включая synthetic transcripts);
- `synthetic-multilingual` — блокирует P1.3 (rune-семантика TF-IDF);
- `synthetic-edge-cases` — блокирует P1.5, P1.7 (corrupt WAL, empty status, duplicate slugs).

**External-01 роль:** локально у мейнтейнера (не коммитится), используется в cross-reference protocol раз в калибровку. Результат cross-reference фиксируется в `docs/fixtures/real-corpus-notes.md` (текст, не данные).
