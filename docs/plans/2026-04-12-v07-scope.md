# v0.7 Scope Draft — Closing the Feedback Loop

**Status:** planning, not started. v0.6.0 released + hotfix `ec9e97b` landed. v0.6 is living in production for observation before v0.7 kicks off.

**Philosophy anchor:** v0.6 started *collecting* signals (trigger-useful/silent, session-metrics, schema-error, outcome-positive/negative). v0.7 starts *reading* them — memory that learns from how it was used.

---

## Core theme

**Close the loop that v0.6 opened.** Every signal we emit to WAL should feed back into future ranking, filtering, or self-diagnostics. If a signal doesn't drive a decision, either we don't need it or v0.7 needs to wire it up.

---

## MVP candidates (in priority order)

### 1. Per-trigger precision & auto-demotion
Currently WAL records `trigger-useful` / `trigger-silent` for every injected slug per session. Nothing reads it.
- Aggregate last N sessions per slug → `useful_ratio`
- Surface low-precision triggers in SessionStart warnings (observability first, not yet auto-action)
- Stretch: triggers with <20% useful over 10+ injections get auto-demoted in priority key (new field `precision_penalty`)
- **Why first:** direct, measurable, closes v0.6 foundation explicitly marked as such in CHANGELOG

### 2. Strategy success_count from WAL
`strategies/*.md` has implicit `success_count`, WAL emits `strategy-used` + `strategy-gap`.
- Aggregator updates `success_count` in frontmatter periodically (stop hook or dedicated cron/script)
- Use `success_count` in strategy ranking — proven strategies surface first
- **Why:** uses existing signal, minimal new surface

### 3. Mistake outcome tracking
WAL has `outcome-positive` (injected mistake + clean session) and `outcome-negative` (same mistake recorded again after being injected — meaning injection didn't prevent repeat).
- Positive/negative ratio per mistake → surfaces which mistakes are being *learned* vs which keep recurring
- Mistakes with high negative ratio → candidates for rewrite (current wording isn't landing)
- **Why:** first-class visibility into whether memory actually changes behavior

### 4. Schema validator in CI/tests
Today's audit found 3 files without frontmatter (`avatar-full-analysis-klipot.md`, `avatar-full-analysis-sephirot.md`, `_agent_context.md`). Runtime tolerates them, but they sit outside ranking entirely.
- `hooks/validate-frontmatter.sh` — exit-code + report for missing required fields by type
- Hook into test-memory-hooks.sh or pre-commit
- **Why:** cheap hygiene, prevents silent exclusion

## Stretch / out-of-scope for v0.7

- **self-profile.md** (original roadmap v0.7): mostly reasoning-layer, generated content. Probably post-v0.7 once precision/success data has accumulated for a few weeks.
- **Meta-patterns detection** (3+ mistakes with similar root-cause → consolidation suggestion): needs real data density, defer.
- **Rule evolution** (rules mutating from stats): conceptually v0.8 territory, depends on v0.7 aggregators.
- **Cross-project transfer** (original roadmap v0.6 name "перенос опыта" — never implemented): still valuable but independent from feedback loop. Park it.

## Technical debt carried from v0.6 audit

Not blocking v0.7, but watch:
- `~/.claude/hooks/lib/wal-lock.sh` and `hypomnema/hooks/lib/wal-lock.sh` share an inode (hardlink or APFS clonefile). A future `cp` or `install.sh` re-run can silently fork them. Action: verify install.sh creates a symlink, add health check for divergence.
- `ref_count` backfill has no reverse consistency check (WAL count vs frontmatter). If someone hand-edits, drift is silent.
- `session-metrics` WAL format was briefly changed and reverted during the WAL locking fix. wal-compact and analytics parsers were not re-verified against the current 4-field layout — smoke them before v0.7 aggregators read the same format.
- `decay_rate` only drives lifecycle rotation in session-stop, not ranking. Documented in CLAUDE.md now; no action needed unless v0.7 aggregators want to use it.

## Observation period

Live on v0.6 for ≥3–5 real working sessions before starting v0.7 implementation. Watch `.wal` for:
- Unexpected duplicates (dedup keys missing an event type)
- `trigger-silent` dominating (noisy triggers to act on in v0.7 MVP #1)
- `outcome-negative` repeats on same mistake (signal for v0.7 MVP #3)
- Schema-error surfacing previously undetected malformed files
- Any health warnings that didn't surface in the synthetic audit

If the live data shows a different signal distribution than we assumed, MVP ordering gets rewritten before implementation starts.

## Open questions (answer before writing specs)

1. **Aggregation cadence** — do we update `useful_ratio` / `success_count` on every Stop hook (cheap but chatty), or batch in a periodic script (cleaner but needs scheduling)?
2. **Auto-action vs observability** — is v0.7 allowed to auto-demote triggers, or only surface them and let the operator decide? Bias toward the latter for MVP; graduate to auto on v0.7.1+ after we trust the metric.
3. **Signal decay** — is a slug's `useful_ratio` computed over all-time WAL or windowed (last 30 days)? Rigid triggers that *become* useful after a context shift deserve a rolling window.
4. **Precision floor for auto-demote** — how many injections before a trigger earns an opinion? Tentative: 10+ injections, <20% useful.
