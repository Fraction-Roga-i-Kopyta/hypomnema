# v0.14 perf measurements (2026-04-24)

Re-running `bash hooks/bench-memory.sh perf` on the HEAD of v0.14
against the baseline `audit-2026-04-16` captured in the bench
script itself. Goal: verify that the three v0.14 additions that
touch hot-path bash (cold-start gate in `compute_wal_scores`,
`.secretsignore` pattern scan in `memory-secrets-detect`,
P1.1 evidence-match pass in Stop) did not regress > 1.5×.

## Hardware

- macOS darwin 25.x, M-series.
- Cold disk cache for the first run of each scenario.

## 500-file synthetic corpus — hot-path latency

| Hook | v0.14 HEAD | Baseline 2026-04-16 | Δ | Guard |
|---|---:|---:|---:|---|
| session-start | 816 ms avg (768 / 873 min/max) | ≈ 700 ms | **+17 %** | PASS (< 1.5×) |
| user-prompt-submit (no-match) | 3787 ms avg (3544 / 4106) | ≈ 3400 ms | +11 % | PASS |
| user-prompt-submit (broad) | 5103 ms avg (5035 / 5186) | ≈ 5000 ms | +2 % | PASS |

### Where the +17 % on session-start comes from

Three candidates, in order of likelihood:

1. **Cold-start gate in `compute_wal_scores`** (P1.0) — the awk pass
   now maintains a second cutoff (`cutoff_gate`, 14-day default)
   alongside the original 30-day `cutoff_30d`, and accumulates
   `gate_count[slug]` in parallel with `pos_count`/`neg_count`.
   Constant-factor ~2× more conditional checks inside the main
   loop, O(WAL size) overall. On a 500-file corpus with synthetic
   WAL the pass still runs in tens of ms; the regression budget is
   mostly absorbed there.

2. **`ref_count` slurp-mode migration** (P1.2) — previous code used
   `perl -pi -e` (streaming) on the injected-files list; v0.14 uses
   `perl -i -0777` (slurp). Per-file overhead goes up, but the list
   is capped at `MAX_FILES + MAX_CLUSTER` (<= 15 in practice), so
   the absolute cost is small.

3. **TF-IDF vocab-count gate** (P1.0) — `wc -l < .tfidf-index`
   runs once per SessionStart. Cheap, but measurable.

Not running a flame graph here — the 17 % is within guard and the
individual contributions are bounded by design. Re-visit if the
next external corpus measures > 1.3× on this scenario.

## Not measured (should be)

- **Stop-hook latency.** v0.14 added a second
  `classify_outcome_positive_from_evidence` pass after
  `classify_feedback_from_transcript`. Both walk the WAL via
  `wal_run_locked awk`, so the overhead is one extra awk
  invocation per Stop. Perf scenario added to `bench-memory.sh`:

  | scenario | min | avg | max |
  |---|---:|---:|---:|
  | stop (3 WAL passes + evidence, 500-file corpus) | 3267 ms | 5031 ms | 8539 ms |

  First v0.14 measurement — no prior baseline. Variance is high
  because Stop does lifecycle rotation, metrics, three WAL scans,
  transcript parse, and reminder assembly in one shot; it's O(corpus-size)
  on the lifecycle pass. Typical user corpora are 50–150 files, so
  wall-clock at that scale should be ~1/3 of these numbers.
  Re-measure after the corpus size normalises in real use.

- **PreToolUse `memory-secrets-detect` on memory paths.** The
  v0.13 detector runs perl once; v0.14 adds a pattern-walk over
  `.secretsignore.default` + `.secretsignore`.

  | scenario | min | avg | max |
  |---|---:|---:|---:|
  | secrets-detect (0 patterns) | 58 ms | 58 ms | 61 ms |
  | secrets-detect (10 patterns) | 58 ms | 59 ms | 61 ms |
  | secrets-detect (100 patterns) | 61 ms | 61 ms | 62 ms |

  Pattern-walk cost is linear but with a negligible constant.
  100 patterns add ~3 ms on top of the baseline 58 ms. Users
  can comfortably carry 500+ patterns before the detector
  noticeably slows Write/Edit tool calls.
- **Cold-start vs mature corpus gate overhead.** Synthetic-mature
  has outcome events; synthetic-cold-start does not. Different
  code paths in `compute_wal_scores`. Single-scenario bench does
  not cover both. Adding as follow-up in the perf suite.

  Ad-hoc micro-bench here (10 files, empty vs 14-day × 5-slug WAL):

  | WAL state | compute_wal_scores run 1 | run 2 | run 3 |
  |---|---:|---:|---:|
  | empty (gate dormant) | 41 ms | 41 ms | 41 ms |
  | mature (gate active) | 41 ms | 44 ms | 41 ms |

  Identical to measurement noise. Awk makes a single pass over the
  WAL regardless of which branch fires in `END{}`; the gate check
  is two extra conditional comparisons per row. So the +17 % on
  the 500-file session-start scenario is distributed across the
  other v0.14 additions (slurp-mode perl for `ref_count`, vocab
  line count, and so on), not concentrated in the gate.

## Regression policy

`bench-memory.sh perf` regression guard says "eyeball current vs
baseline; investigate > 1.5× slowdown". v0.14 is at +17 % / +11 %
/ +2 % — within budget, no action required.

CI integration for this benchmark is still blocked on a
percentile-vs-baseline threshold policy (flagged repeatedly in
CHANGELOG since v0.10.2). That is a v0.15+ task.

## Baseline update

Bench script `hooks/bench-memory.sh` still prints the
audit-2026-04-16 line as the baseline. Not updating in this
release — the +17 % is real but small, and keeping the original
line visible makes the drift obvious if a future change crosses
the 1.5× threshold. When v0.15 lands and either adds more hot
paths or intentionally re-baselines, update the printed line and
reference this measurement file.
